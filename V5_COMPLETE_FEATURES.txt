✅ Unsupervised ML Clustering Engine
───────────────────────────────────
Feature: Detect coordinated multi-source attacks using multi-dimensional clustering

Algorithm: K-means style clustering with adaptive distance metrics

Features:
  ├─ Multi-dimensional distance calculation
  ├─ Automatic cluster detection (no pre-specified K)
  ├─ Coordinated attack identification
  ├─ Temporal correlation analysis
  └─ Pattern-based threat classification

Multi-Dimensional Analysis:
  Dimensions analyzed:
    1. Packet Rate (pps) - Normalized 0-1
    2. Risk Score (0-100) - Normalized 0-1
    3. Temporal Proximity (hours) - Recent attacks cluster
    4. Tool Type Match - Same tool = closer cluster
  
  Euclidean Distance Formula:
  distance = √[ (pps_diff/max_pps)² × 0.3 +
                (risk_diff/100)² × 0.2 +
                (time_diff_hours)² × 0.3 +
                (tool_mismatch)² × 0.2 ]

Clustering Process:
  1. Add attack vectors to clustering engine
  2. Calculate pairwise distances between all attacks
  3. Group attacks within max_distance threshold (0.4)
  4. Identify coordinated attacks (3+ IPs in cluster)
  5. Classify coordination pattern

Attack Coordination Detection:
  ├─ Single-source attack: 1 IP, 1+ tools
  ├─ Distributed attack: 3+ IPs, same tool
  ├─ Multi-tool attack: 1+ IPs, multiple tools
  └─ Coordinated botnet: 10+ IPs, synchronized timing

Code Location: ClusteringEngine class (lines 116-203)

Example Cluster Detection:
  
  Attack 1: 192.168.1.100, HULK, 500 pps, Risk 85%, 14:00
  Attack 2: 192.168.1.101, HULK, 480 pps, Risk 84%, 14:02
  Attack 3: 192.168.1.102, HULK, 510 pps, Risk 86%, 14:01
  
  → Detected as COORDINATED BOTNET (3 attackers, 1483 total pps)

3️⃣  REAL-TIME AUTONOMOUS MITIGATION (CLOSING THE LOOP)
═════════════════════════════════════════════════════════════════════════════════

✅ Native Subprocess System Call Execution
──────────────────────────────────────────
Feature: Direct iptables integration for real-time attack mitigation

Mitigation Actions:

A) Hard Block (Complete Drop)
   iptables -I INPUT -s <source_ip> -j DROP
   • Blocks all packets from attacker
   • Immediate effect (no packet processing)
   • Default duration: 1 hour (configurable)
   • Ideal for: HOIC, LOIC, sustained attacks

B) Rate Limiting (Partial Acceptance)
   iptables -A INPUT -p tcp -s <ip> -m limit --limit <pps> -j ACCEPT
   iptables -A INPUT -p tcp -s <ip> -j DROP
   • Allows up to X packets per second
   • Drops anything above threshold
   • Better for: TCP-based slowloris, R.U.D.Y
   • Preserves legitimate traffic

C) Selective Blocking (Port-Specific)
   iptables -I INPUT -p tcp -s <ip> --dport 80 -j DROP
   • Block only web traffic
   • Preserve other services
   • Useful for: HULK (HTTP-specific)

Implementation Architecture:

AutonomousMitigation Class:
  ├─ block_ip(source_ip, reason, duration)
  ├─ unblock_ip(source_ip)
  ├─ rate_limit_ip(source_ip, limit_pps)
  ├─ get_mitigation_log(limit)
  └─ blocked_ips tracking set

Execution Flow:
  1. Detection engine identifies attack
  2. CVSS score calculated
  3. Risk threshold exceeded?
       ├─ Yes: Trigger autonomous mitigation
       │   ├─ Block IP via subprocess
       │   ├─ Log action to database
       │   ├─ Schedule auto-unblock
       │   └─ Update UI in real-time
       └─ No: Alert-only (no blocking)
  4. Unblock timer triggers after duration
  5. Verify removal via iptables -L

Code Location: AutonomousMitigation class (lines 211-301)

Safety Features:
  ├─ Root privilege verification (os.geteuid() == 0)
  ├─ Dry-run mode (test without actual blocking)
  ├─ Timeout on subprocess execution (5 seconds max)
  ├─ Error handling for failed iptables calls
  └─ Automatic logging of all actions

Configuration:
  # Enable/disable autonomous mitigation
  AUTO_MITIGATION_ENABLED = True  # Set to False to disable
  
  # Dry-run mode (test mode)
  mitigation = AutonomousMitigation(enabled=True, dry_run=True)
  
  # To enable actual blocking:
  mitigation = AutonomousMitigation(enabled=True, dry_run=False)

Monitoring:
  Dashboard Tab: "🛡️ Mitigation Log"
  ├─ Blocked IP count (real-time)
  ├─ Recent actions with timestamps
  ├─ Block durations
  ├─ Success/failure status
  └─ Automatic expiration tracking

Example Mitigation Timeline:

✅ Comprehensive CVSS 3.1 Risk Scoring
────────────────────────────────────
Feature: Standards-based vulnerability severity rating for DDoS attacks

CVSS Components Calculated:

A) Base Score Components:
   
   Attack Vector (AV):
   ├─ Network (0.85) - Exploitable from network
   ├─ Adjacent (0.62) - Requires network adjacency
   ├─ Local (0.55) - Requires local access
   └─ Physical (0.20) - Requires physical access
   
   Attack Complexity (AC):
   ├─ Low (0.77) - No special conditions
   ├─ High (0.44) - Special conditions required
   
   Privileges Required (PR):
   ├─ None (0.85) - No privileges
   ├─ Low (0.62) - Low privileges
   ├─ High (0.27) - High privileges
   
   User Interaction (UI):
   ├─ None (0.85) - No user action required
   ├─ Required (0.62) - User must interact
   
   Confidentiality (C):
   ├─ High (0.56) - All data disclosed
   ├─ Low (0.22) - Limited data exposure
   ├─ None (0.00) - No impact
   
   Integrity (I):
   ├─ High (0.56) - Data can be modified
   ├─ Low (0.22) - Limited modification
   ├─ None (0.00) - No impact
   
   Availability (A):
   ├─ High (0.78) - Complete unavailability
   ├─ Low (0.27) - Degraded performance
   ├─ None (0.00) - No impact

B) Temporal Factors:
   ├─ Exploit Code Maturity (mature = higher)
   ├─ Attack Complexity increase (higher = lower score)
   └─ Exploit Publication Status (published = higher)

C) Environmental Factors:
   ├─ Threat Intelligence availability
   ├─ Detection method effectiveness
   ├─ Recovery difficulty
   └─ Remediation availability

CVSS Score Ranges:

  0.0       → No Impact
  0.1-3.9   → Low (Green)
  4.0-6.9   → Medium (Yellow)
  7.0-8.9   → High (Orange)
  9.0-10.0  → Critical (Red)
